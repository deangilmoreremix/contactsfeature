# Plan: Diagnose & Fix Blank Production App (SmartCRM Contacts Remote)

Goal: Restore a working production build at contacts.smartcrm.vip where the app loads its full UI (standalone and via host) and eliminate the blank page with no console errors.

Constraints:
- This planning step is read-only; no edits or execution beyond writing this plan file.
- Implementation steps below will be executed only after approval.

Summary of current state and issues:
- App previously worked; after Module Federation + Vite 8/plugin upgrades and debugging changes, the live site is blank with an empty #root and no console errors.
- We added debug markers (main.tsx, index.html, SmartCRMApp) and dependency fixes (zod, @vitejs/plugin-react) and Netlify config (NPM_FLAGS).
- Netlify builds succeed, but served page appears to load older assets or otherwise not execute the new bundle (no debug logs in console). Browser shows extension logs (content_topFrameLifeline) that obscure output.

Success criteria (acceptance):
1. The app loads the UI on direct visit to contacts.smartcrm.vip (standalone) and when embedded by the host (app.smartcrm.vip).  
2. Console shows expected debug markers during verification; no uncaught runtime errors.  
3. Netlify deploys succeed with Functions bundling; no unresolved imports like zod/v4.  
4. After fix, debug markers and verbose logs are removed and normal prod settings restored.

Plan phases (high level):
A. Stabilize local reproducible build and confirm debug markers exist in built output.
B. Diagnose why live site serves blank page (asset caching vs runtime failure).
C. Fix root cause (cache invalidation, service worker, runtime code error) and verify locally.
D. Deploy cleanly to Netlify (clear cache deploy), verify, then remove debug artifacts.

Detailed steps (ordered, with commands to run during implementation):

PHASE A — Local verification & reproducible build
1. Ensure working tree clean and on `main` branch (confirmed).  
2. Reproduce build locally and confirm debug markers included:
   - rm -rf dist && npm ci --legacy-peer-deps && npm run build
   - Verify dist/index.html and dist/assets/index-*.js contain markers: "BUILD TIMESTAMP", "JS EXECUTED", "BUILD_ID".
   - Serve locally: npx serve dist -l 5176 (or npm run preview) and open http://localhost:5176.
   - Confirm red debug banner and console logs appear locally.
   - If local build shows blank as well, proceed to debug module execution locally (see PHASE C). If local works but Netlify blank, skip to PHASE B.

PHASE B — Diagnose live site behavior (Netlify)
1. In production, open DevTools → Network. Enable "Disable cache". Hard reload (Ctrl+Shift+R).
2. Inspect /index.html response body: confirm presence of inline bootstrap marker and correct script tag. (If not, Netlify didn't deploy the latest build.)
3. Inspect the response body of the listed script tag (the hashed index-*.js). Search for debug markers (BUILD_ID, DEBUG). If not present, CDN/cache serving old assets.
4. Check Service Workers: DevTools → Application → Service Workers. If registered, unregister and clear cache storage. Reload.
5. Filter console logs to exclude noisy extension logs (e.g., filter out content_topFrameLifeline). Then capture first 20 console lines.
6. On Netlify dashboard, check the deploy used commit SHA; ensure it matches `git rev-parse main` at the time of the push. If mismatch, re-trigger clear cache & deploy.

PHASE C — If runtime code failure (local or production) after debug markers present
1. Make a minimal reproduction: replace SmartCRMApp render with a trivial fallback component to test boot:
   - Create `src/MinimalRoot.tsx` that renders a simple div and export it as SmartCRMApp via temporary swap.
   - Build + serve locally and verify minimal root loads. If it loads, the issue exists inside SmartCRMApp or its top-level imports.
2. Narrow down: progressively comment out or lazy-load heavy top-level imports in SmartCRMApp (e.g., large pages, AI contexts, services) until the app boots. Use binary search to find failing import or code.
3. Check for top-level side-effects — any module that runs during import and expects environment variables or browser APIs may throw silently in the build environment. Replace `import.meta.env` with safe accessors for server-side code (process.env fallback) — we already changed contactAgentSettings.
4. Check for other bundler-specific incompatibilities (e.g., plugin-federation interactions, virtual CSS IDs). If errors reference these, revert to minimal build and reintroduce patches (e.g., Vite 8 CSS workaround plugin) carefully.

PHASE D — Fix, verify locally, and deploy
1. Once root cause fixed and local build works, run `npm run build` and verify built artifacts contain no debug markers (remove them before final push).  
2. Commit changes with a clear message and push to `main`.
3. On Netlify: trigger **Clear cache and deploy site**. Monitor deploy logs; verify functions bundling and assets upload.  
4. After deploy succeed, hard-refresh the site and verify UI loads; inspect console and network.
5. If successful, remove any temporary Netlify `NPM_FLAGS=--legacy-peer-deps` or other workarounds if not desired permanently.

PHASE E — Postmortem & cleanup
1. Remove debug markers from source (main.tsx, index.html, SmartCRMApp).  
2. Re-enable console dropping in Vite config if desired (or keep log preservation).  
3. Add small automated checks (smoke test): a short script that hits the deployed URL, checks presence of a known DOM element (e.g., element with data-smartcrm-remote-root), and fails if blank. Hook into CI or Netlify post-deploy webhook.
4. Document the root cause and the fix in a short entry in `MF_REMOTE_STANDALONE_LANDING_FIX_COMMIT_DOCUMENTATION.md` and in the `NETLIFY_DEBUG_NOTES.md` file.

Rollback plan
- If a change causes production regression, revert the commit that introduced the regression and re-deploy immediately:
  - git revert <sha> && git push origin main → Netlify should rebuild from the reverted commit.

Risk assessment & mitigations
- Risk: Netlify serving stale assets from CDN even after deploy. Mitigation: force different asset filenames by changing build output (we added BUILD_ID) and use "Clear cache and deploy".
- Risk: Functions bundler failing on node-only imports. Mitigation: ensure server modules use process.env and are not importing browser-only features. Keep function dependencies listed in top-level package.json.
