Phase 2 — Quick remediations (fast experiments, non-invasive)

Phase 3 — Diagnose runtime/module-execution if issue persists
- If local reproduction fails (works locally) but live fails → CDN/Edge/SW or HTML/script mismatch
- If local reproduction also fails → code-level runtime failure
  Key diagnostic checks:
  - Check for top-level exceptions in main module (syntax error, runtime error) visible in console. Capture full stack traces.
  - Check Module Federation ordering: ensure remoteEntry is loaded before consumer tries to import exposed modules. If consumer executes before remote registers, the loader can stall.
  - Check script attributes: index.html currently loads index-BLpO_U3d.js with `type="module" crossorigin` (correct for ESM). Ensure any bootstrap which dynamically loads remoteEntry uses the right import() or script element with type="module" as needed.
  - Confirm remoteEntry registers expected exposes (SmartCRMApp) and that E() import wrapper resolves to module default export. Inspect remoteEntry.js to ensure exports and dynamic loader E(e) function are correct.

Phase 4 — Fixes (repo changes, medium confidence)
- Short-term (minimal code change)
  1. Move inline bootstrap into a small hashed external file (e.g., public/bootstrap.js or src/bootstrap.ts → build emits /assets/bootstrap-<hash>.js). Rationale: prevents stale inline script being cached by CDNs/HTML and forces index.html change on deploy.
  2. Add stronger runtime logging in bootstrap and in main entry (console.log at start, before/after remote init). Add try/catch around top-level await/import to surface errors into the inline UI.
  3. Ensure the bootstrap explicitly loads remoteEntry before starting the app when using MF (dynamic script loader that returns a Promise). Example pattern:
     - create script element for remoteEntry.js (module or nomodule depending on build)
     - await on load event
     - call __webpack_init_sharing__ / init (if required by plugin) then dynamically import or call get("./SmartCRMApp")
- Long-term (robustness)
  - Version Service Worker cache name using build timestamp (CACHE_NAME = `contacts-app-${BUILD_ID}`) so SW invalidation occurs on each release.
  - Embed build-id or timestamp metadata into index.html (meta name="x-build-id") to detect stale clients and force refresh UI if mismatch.
  - Add CI verification: after build, compute and compare artifact hashes against last deployed commit; if hashes are identical, optionally force a small dummy change in dist (or fail the deploy and require manual confirmation).
